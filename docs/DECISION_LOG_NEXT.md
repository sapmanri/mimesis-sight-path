### 조사 결과 — 겹침은 재현되는 실제 버그

카드 레이아웃 상수를 실측한 결과, **취향 7개 이상에서 footer를 침범**한다.

| 상수 | 값 |
|---|---|
| 카드 높이 H | 1350 |
| 스냅샷 | y=386..686 |
| 수치 baseline | y=748 |
| 목록 시작 | y=844 |
| 행 높이 | 64 |
| footer 텍스트 | y=1278 |

| 행 수 | 마지막 행 bar 하단 | 결과 |
|---|---|---|
| 5 | 1159 | 정상 |
| 6 | 1223 | 정상 |
| 7 | 1287 | **footer 침범** |
| 8 | 1351 | **카드 밖으로 나감 (H=1350 초과)** |

현재 코드는 `this.rows().slice(0,8)`로 최대 8개를 그리므로, **취향 7개 이상인 관찰자는 항상 깨진 카드를 받는다.** 생성된 카드 이미지에서 footer 문구가 목록 행 위에 겹쳐 판독 불가한 것을 육안으로도 확인했다.

원인은 413-A 구현 시 "대표 5~8개"라는 요구를 상한만 지키고 **세로 예산을 검증하지 않은 것**이다. 지시서의 "5~8개"는 안전 범위가 아니라 사실상 6개까지만 안전하다.

### 영향 범위
- `PR.buildCard()` 한 함수. 다른 경로 영향 없음.
- 모달 시트(`PR.render`)는 내부 스크롤이라 무관 — 카드만의 문제.

### 설계 제안 (구현 전 승인 필요)
1. **표시 개수를 세로 예산에서 역산**해 상수로 고정 (footer 위 여유 24px 확보 시 최대 6행).
2. 초과분은 목록 끝에 `외 N개` 한 줄로 요약.
3. 코드 마스킹: `BYL-A7K2-91QF` → `BYL-A7K2-••••` 형태. 단 **P0-2 결론에 따라 마스킹만으로 불충분할 수 있음**(아래 참조).

## [P0-2] 복구 진입점 — 구조 조사

### 요구
- 저장된 코드로 기존 Byeol을 복원하는 입력 UI가 현재 없음
- 첫 실행/설정에 "기존 Byeol 이어가기" 진입점 필요
- Observer Code가 공개 식별 / 공동 연결 / 계정 복구를 동시에 하는지 구조 조사
- 하나의 코드가 복구 권한까지 가진다면 공개 ID와 Recovery Key 분리 설계안을 먼저 보고

### 조사 결과 — **현재 Observer Code는 복구 권한이 없다**

코드 전수 조사 결과, `observerId`의 실제 역할은 **표시 전용**이다.

`observerId` 사용처 (전부):
| 위치 | 용도 |
|---|---|
| `_empty()` | 생성 |
| `load()` | 존재 여부 검증만 (`p.observerId` truthy 체크) |
| `copyCode()` | 클립보드 복사 |
| `PR.render()` | 시트 헤더 표시 |
| `buildCard()` | 카드에 그리기 |

**결정적 사실 2가지:**

1. **서버는 observerId를 전혀 모른다.** `functions/`, `authority/src/`, `src/` 어디에도 `observerId`/`observer_id` 참조가 없다. `live-sync.js`는 `/api/byeoli/state`로 **GET만** 하며 관찰자 식별자를 보내지 않는다. Authority는 관찰자를 식별하지 않는다.

2. **코드로 데이터를 불러오는 경로가 존재하지 않는다.** `load()`는 오직 `localStorage[PROFILE_KEY]`만 읽는다. 코드를 입력받아 프로필을 조회하는 함수·엔드포인트가 없다.

**결론: 현재 Observer Code는 "공개 식별자"일 뿐이며, 복구·인증 권한이 없다.** 남에게 코드가 노출돼도 그 사람이 내 기록을 가져갈 수 없다. 데이터는 브라우저 localStorage에만 존재하고, 코드는 그 데이터를 가리키는 열쇠가 아니다.

따라서 **지금 시점에서는 P0-1의 마스킹만으로 충분하다.** "공유 가능한 코드가 복구까지 가능하다면 마스킹으로 해결되지 않는다"는 우려는 정확한 지적이지만, 현재 구조에서는 아직 성립하지 않는다.

### 다만 — 서버 저장으로 옮기는 순간 성립한다

413-A는 "이후 서버 저장으로 옮길 수 있도록 Profile 객체로 분리"를 명시했다. 그 이관 시점에 `observerId`가 조회 키가 되면, **그때부터 공개 코드 = 복구 키**가 되어 지적하신 문제가 현실이 된다.

추가 위험: 현재 코드 엔트로피는 **39.6비트**(31자 알파벳 × 8자 = 8.5×10¹¹). 공개 식별자로는 충분하지만, **복구 키로 쓰기에는 부족하다** — 온라인 무차별 대입에 취약하고, 카드에 인쇄되어 유포된다는 점에서 더욱 그렇다.

### 권고 설계 — 공개 ID와 Recovery Key 분리

서버 이관 **이전에** 다음 구조를 확정할 것을 권고한다.

| | Observer Code (공개) | Recovery Key (비밀) |
|---|---|---|
| 역할 | 표시·공유·공동 연결 | 계정 복원 인증 |
| 노출 | 카드·시트에 표시 가능 | 절대 카드/시트에 표시 금지 |
| 형식 | `BYL-A7K2-91QF` (39.6bit) | 별도 고엔트로피 (≥128bit 권장) |
| 저장 | 평문 | 서버에는 해시만 |
| 재발급 | 불가(정체성) | 가능 |

이 구조라면 카드에 코드가 그대로 실려도 복구 권한이 새지 않으므로, P0-1 마스킹은 "프라이버시 보호"(누가 내 코드를 수집하는 것 방지) 목적으로만 남는다.

### 영향 범위
- **현재 구현 변경 없음** — 조사만 수행.
- 진입점 UI("기존 Byeol 이어가기")는 **복구 백엔드가 없으면 만들 수 없다.** 지금 UI만 만들면 입력해도 아무 일도 일어나지 않는다. 서버 이관과 반드시 같은 빌드에서 진행해야 한다.
- 서버 이관 시 영향: `Profile.load/save`, 신규 인증 엔드포인트, Authority(현재 관찰자 무지 상태) 설계 변경.

### 다음 결정 필요 사항
1. 서버 이관을 언제 할 것인가 — 그 전까지 복구 UI는 보류.
2. 이관 시 위 2키 분리 구조를 채택할 것인가.
3. 채택 시 기존 관찰자(코드만 가진 사용자) 마이그레이션 방법.
